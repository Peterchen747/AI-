### 核心診斷結論
> **現在這個工具算的不是「淨利」，是「不完整的毛利」。**
> 因為：(1) 成本是手動輸入，沒有從進貨批次自動帶入；(2) 廣告費、運費、包材從來沒有被計入。
> 在修復這兩個問題之前，圖表上的數字對商業決策沒有意義。

## PART 1：產品定位框架

### 一句話定位
> 台灣飾品類聊天成交賣家的**每週 15 分鐘經營體檢工具**——讓「靠感覺在飛」的小賣家，第一次看到自己真實的月度淨利。

### 不是什麼（同樣重要）
- ❌ 不是一般記帳軟體（Moze、記帳城市）
- ❌ 不是電商後台（Shopline、91APP）
- ❌ 不是 ERP 或進銷存系統
- ❌ 不是稅務工具
- ✅ 是「聊天成交 → 真實淨利」這條路徑上的唯一工具

### 核心價值主張（依優先順序）
1. **我知道我這個月真正賺了多少**（vs. 現在靠猜）
2. **我知道廣告和運費吃掉我多少**（隱藏成本可視化）
3. **我每週只花 15 分鐘就能完成帳務整理**（低摩擦習慣）

## PART 2：用戶框架

### 種子用戶（N=1 驗證期）
**畫像**：女性，台灣，賣玉髓/飾品，月營收約 10 萬，IG/LINE 接單，銀行轉帳收款，截圖存相簿，目前靠戶頭餘額判斷賺賠，一天 3-5 筆訂單，希望每週批次處理一次帳務。

**她現在的工作流**：
```
客戶 IG 私訊 → 確認品項金額 → 客戶轉帳截圖 → 確認戶頭入帳 → 安排出貨
                                                              ↓
                                             （帳務這段現在靠感覺）
```

**她需要的轉變**：
```
每週日晚上 → 打開工具 → 15 分鐘確認本週訂單與成本 → 看到本週淨利 → 關掉
                                                              ↓
                                             月底看到完整月度報告
```

### 第二階段目標用戶（5-10 人驗證期）
- 台灣，同類型飾品/手工藝/選物類賣家
- 月營收 NT$5 萬 – 30 萬
- 同樣以聊天平台成交、銀行轉帳收款
- **獲取管道**：種子用戶（老婆）的同行圈口碑

### 商業化目標用戶（付費期）
- 第一垂直：台灣飾品類賣家（TAM 估計 5,000-20,000 人）
- 第二垂直（未來）：台灣手工藝/選物/手工食品類賣家
- **付費轉換假設**：TAM 的 2-5% = 100-1,000 付費用戶

### 永遠不服務的用戶
- 月營收 100 萬以上（需要正式會計）
- 有自架電商後台的賣家
- 不願意每週花時間整理帳務的賣家

## PART 3：核心工作流框架

### 主工作流：每週體檢（核心中的核心）

```
觸發點：每週日晚上的推播通知「本週有 N 筆待確認訂單」
                    ↓
Step 1：確認本週訂單（每筆 20 秒）
  - 選商品（從庫存清單下拉）
  - 填售價 ✓ 確認已收款 ✓
  - 系統自動扣庫存、自動算這筆毛利
                    ↓
Step 2：登記本週成本（2 分鐘）
  - 廣告費 / 運費 / 包材 / 其他
  - 一個週期一個畫面，不拆細
                    ↓
Step 3：看結果（30 秒）
  - 本週淨利 ＝ 毛利總和 - 本週成本
  - 月累計淨利進度條
  - 有無異常警示
```

### 次工作流：進貨登記（不定期）
```
新進一批貨 → 新增進貨批次
  - 品項名稱
  - 進貨數量
  - 進貨總金額
  - 系統自動算單件成本
  → 之後訂單選這個品項時自動帶入成本
```

### 次工作流：月底複查（每月一次，5 分鐘）
```
月底 → 系統自動生成月度報告
  - 本月營收 / 總成本 / 淨利 / 淨利率
  - 成本結構圓餅圖（廣告 / 運費 / 進貨 / 其他）
  - 與上月比較
  - 警示：哪個成本項目異常升高
  → 可匯出 PDF
```

## PART 4：資料模型框架

### 現有 Schema vs. 需要的 Schema 對照

```
【現有 - 保留】categories 表 ✅
  id, name, description, isActive, createdAt, updatedAt

【現有 - 保留】items 表 ✅（微調）
  id, categoryId, name, typicalCost（參考用，不再是主要成本來源）,
  typicalPrice, isActive, createdAt, updatedAt

【需要新增】purchaseBatches 表（真正的進貨批次）────────────
  id                主鍵
  itemId            對應 items 表（外鍵）
  purchaseDate      進貨日期
  totalQty          進貨總數量
  remainingQty      目前剩餘庫存（初始 = totalQty，每賣 -1）
  totalCost         這批進貨的總金額（NT$）
  unitCost          單件成本（totalCost / totalQty，系統自動計算）
  notes             備註（可空）
  createdAt         建立時間

【現有 - 修改】sales 表 ⚠️
  現有欄位保留，新增：
  purchaseBatchId   外鍵，指向 purchaseBatches（可空，舊資料相容）
  ※ 當 purchaseBatchId 有值時，cost 自動從 purchaseBatches.unitCost 帶入
  ※ 確認銷售後，對應 purchaseBatches.remainingQty 自動 -1

【需要新增】weeklyCosts 表（每週營運成本）────────────────
  id                主鍵
  weekLabel         年份-週數（e.g. "2026-W15"）
  adCost            廣告費（Meta / IG / 其他）
  shippingCost      運費支出（賣家自負部分）
  packagingCost     包材費用
  otherCost         其他雜支
  totalCost         系統自動加總
  notes             備註（可空）
  createdAt         建立時間

【現有 - 降優先級】importBatches / screenshots 表
  重新命名為 ocrImportBatches 避免與進貨批次混淆
  功能保留但不再是核心，v2 再優化

【現有 - 保留】shareTokens 表 ✅
```

### 關鍵計算邏輯（修正後）

```
每筆銷售毛利 = actualPrice - purchaseBatch.unitCost
（不再是手動輸入 cost，而是自動從進貨批次帶入）

月毛利總和 = 當月所有 sales.grossProfit 加總

月總營運成本 = 當月所有 weeklyCosts.totalCost 加總

月淨利 = 月毛利總和 - 月總營運成本   ← 這才是真實淨利

淨利率 = 月淨利 / 月營收
```

### 遷移策略（不破壞現有資料）
- 舊的 `sales.cost` 欄位保留，歷史資料繼續有效
- 新資料透過 `purchaseBatchId` 自動帶入成本
- 系統判斷邏輯：`purchaseBatchId != null` 用批次成本；否則用手動 `cost`
- 這樣舊資料不需要重新輸入，新資料自動更準確

## PART 7：商業化框架

### 定價模型
| 方案 | 價格 | 限制 |
|------|------|------|
| 免費版 | NT$0 | 每月最多 30 筆訂單，不開放 PDF 匯出、警示功能 |
| 專業版 | NT$299/月 | 無限制，全功能 |
| 年繳方案 | NT$2,699/年（省兩個月） | 同專業版 |

### 商業化時間軸
```
現在
 │
 ├─ 第 1-2 個月：補強現有工具到「商業化可展示」水準
 │   → 確保 P0 功能全部正確
 │   → 手機 UI 優化
 │   → 推播通知實作
 │
 ├─ 第 3-4 個月：種子用戶驗證
 │   → 找到 5-10 個同行試用
 │   → 蒐集真實反饋
 │   → 確認「願意付費」的意願
 │
 ├─ 第 5-6 個月：商業化上線
 │   → 帳號系統 + 付費方案
 │   → 第一批付費用戶目標：10 人
 │
 └─ 第 7-12 個月：擴大垂直
     → 從飾品類擴展至手工藝/選物類
     → 目標付費用戶：100 人
```

### 獲客漏斗
```
曝光（IG 飾品賣家社群內容）
    ↓
試用（免費版，30 筆訂單）
    ↓
啟動（完成第一次週度體檢）
    ↓
留存（連續 4 週主動回訪）
    ↓
付費（超過 30 筆訂單限制）
```

### 最重要的獲客洞察
付費行為的最自然觸發點是「免費版的 30 筆訂單限制到頂」。不要讓人因為「覺得好用」才付費——要讓人因為「不付費就不能繼續用」而付費。這是 SaaS 最乾淨的轉換邏輯。

| 風險 | 嚴重度 | 應對方式 |

## PART 10：開發執行順序（框架確認後的具體任務）

> 原則：先把「真實淨利計算」這條主線打通，再做體驗優化，最後做商業化基礎設施。

### 第一階段：打通真實淨利計算（估計 3-4 週）

```
Week 1：資料模型升級
  □ 新增 purchaseBatches 資料表（Drizzle schema migration）
  □ 新增 weeklyCosts 資料表
  □ sales 表新增 purchaseBatchId 欄位
  □ 舊資料相容性測試

Week 2：進貨批次管理頁面
  □ 進貨批次 CRUD（新增/編輯/刪除）
  □ 庫存剩餘數量自動計算與顯示
  □ 低庫存提醒邏輯（< 3 件）

Week 3：銷售流程重構
  □ 銷售輸入時改為「選進貨批次」而非手動輸入成本
  □ 確認銷售後自動扣減庫存
  □ 每筆毛利自動計算

Week 4：每週成本登記 + 淨利計算修正
  □ 每週成本輸入頁面（廣告/運費/包材/其他）
  □ 首頁 KPI 修正：月淨利 = 月毛利 - 月營運成本
  □ 六個月圖表改為顯示淨利（非毛利）
```

### 第二階段：體驗提升（估計 3-4 週）

```
  □ 隱藏成本警示頁面
  □ 手機 RWD 重新審視主要操作流程
  □ 月度報告 PDF 匯出
  □ 每週 Email 提醒（「本週有 N 筆待確認」）
  □ 修正：importBatches 更名為 ocrImportBatches
```

### 第三階段：商業化基礎設施（估計 4-6 週）

```
  □ 資料庫遷移：SQLite → Turso（雲端 LibSQL，相容現有 Drizzle 設定）
  □ 用戶帳號系統（NextAuth.js + Email 登入，最簡方案）
  □ 多租戶資料隔離（每個用戶只看自己的資料）
  □ 付費方案管制（免費 30 筆訂單/月限制）
  □ 付費整合（綠界科技，台灣最適合）
  □ 上線 Vercel + 自訂網域
```

### 不在任何階段的事（現在不碰）

- Meta 廣告 API 串接
- 銀行帳單自動對帳
- 任何「看起來很酷」但跟真實淨利無關的功能

## 附錄：核心設計原則（每次做決定前先看這裡）

1. **每個功能問一個問題**：「這個功能讓用戶更快知道真實淨利嗎？」如果不是，不做。
2. **每次輸入越少越好**：目標是用戶每週最多花 15 分鐘，超過這個就重新設計。
3. **先讓老婆愛上，再讓陌生人買單**：任何功能如果老婆說「這個沒用」，刪掉。
4. **垂直優先，通用延後**：飾品賣家做到滿意再說其他的。
5. **留存比獲客重要**：一個付費用戶用了 12 個月，比 10 個試用就走的用戶有價值。
