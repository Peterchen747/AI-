# 對話式訂單財務助手 PRD

## 產品定位
這是一個給「在聊天室內接單、人工核對匯款、成本散落各處」的小型賣家使用的財務整理工具，核心價值是把分散的資料來源統整成可看、可追、可警示的月度淨利視圖。[cite:139][cite:147][cite:151]

使用情境是：客戶先在私訊或聊天室詢問，下單後轉帳並回傳截圖，賣家再人工比對銀行紀錄；同時，成本又分散在 Meta 廣告、物流/運費、包材平台、以及紙本稅單等不同來源，導致真實淨利不容易即時看見。[cite:147][cite:149][cite:151]

## 核心問題
目前訂單資料、付款證明、銀行入帳、廣告支出、物流成本、包材採購與稅務資料彼此分離，容易讓經營者只看到營收，卻看不到真實淨利與隱藏成本。[cite:147][cite:149][cite:150]

電商與小商家常見的隱藏成本包括支付手續費、運費、退貨、軟體訂閱、包材與稅務等，若沒有自動化整理與分類，毛利與淨利會被高估。[cite:147][cite:148][cite:150]

## 產品目標
產品要做到三件事：

- 不管客戶上傳什麼資料型態，都能先集中收進同一個工作台，再轉成可分類資料。[cite:152][cite:155]
- 支援 Excel 匯入與手動輸入，降低導入門檻，讓原本靠聊天做單的商家也能開始整理帳務。[cite:151][cite:156]
- 直接提供「本月淨利總覽」與「隱藏成本警示頁」，讓使用者先看結果，再往下查原因。[cite:149][cite:150][cite:151]

## 目標客群
第一階段目標客群是以私訊、社群、LINE、IG、Facebook 或熟客聊天方式成交的小型賣家，尚未建立完整 ERP 或電商後台，自行用銀行轉帳、截圖、Excel 與人工對帳管理業務。[cite:139][cite:149]

這類使用者通常不是沒有資料，而是資料分散、格式混亂、來源太多，所以更需要「彙整」而不是先追求複雜會計系統。[cite:147][cite:151][cite:152]

## 使用者流程
### 訂單與收款
1. 客戶在聊天室私訊下單。
2. 商家確認品項、金額、運費與收件資訊。
3. 客戶轉帳後回傳截圖。
4. 商家人工核對銀行入帳。
5. 商家安排出貨並記錄物流與包材成本。

這代表系統的最小可行功能不是「商城前台」，而是「接住聊天交易後的內帳流程」。[cite:139][cite:151]

### 成本整理
成本來源會分散在 Meta 廣告、物流平台、包材購買平台與稅務文件，系統需要能把這些來源都匯入到同一個成本池，再依月份、訂單、渠道與類別歸檔。[cite:147][cite:151]

## MVP 功能範圍
### 1. 通用資料收納箱
支援上傳多種檔案與資料型態，第一版先聚焦在：圖片截圖、PDF、Excel、CSV 與文字貼上。OCR 與發票/單據辨識工具已普遍用於把非結構化文件轉成可用欄位，適合作為此產品的核心能力之一。[cite:152][cite:155]

系統在上傳後不一定一次就要全自動記帳，但至少要先做到：抽取關鍵欄位、讓使用者確認、再存成統一格式資料列。[cite:152]

### 2. Excel 匯入與手動輸入
支援既有 Excel 檔直接匯入，讓使用者不用改掉原本習慣；同時提供手動輸入表單，給零散資料快速補登。[cite:151][cite:156]

手動輸入至少應包含：日期、訂單編號/客戶識別、收入金額、付款方式、物流成本、包材成本、廣告成本、稅務成本、備註。[cite:149][cite:156]

### 3. 自動分類與關聯
系統要將資料分成兩大主軸：

- 收入事件：訂單、轉帳、退款。
- 成本事件：廣告、物流、包材、平台費、稅金、其他雜支。[cite:147][cite:148][cite:150]

若能建立基本關聯規則，例如同日期、同金額、同客戶名稱或同訂單號的配對，就能逐步從人工核帳走向半自動核帳。[cite:152][cite:155]

### 4. 本月淨利總覽
首頁只看最重要的數字：

- 本月營收
- 本月已確認收款
- 本月總成本
- 本月淨利
- 淨利率
- 尚未對帳金額
- 尚未分類成本

即時財務儀表板能幫商家快速看到收入、支出與利潤變化，這是小型電商記帳工具的核心價值之一。[cite:149][cite:151]

### 5. 隱藏成本警示頁
此頁面專門抓那些容易被忽略的成本，至少先做以下警示：

- 廣告花費快速上升，但本月營收或淨利沒有同步成長。[cite:149]
- 物流成本占營收比例異常升高。[cite:148][cite:151]
- 包材/雜支零散累積超過門檻。[cite:147][cite:150]
- 稅務成本尚未登錄，但本月已有大量收入。[cite:147]
- 有收入但找不到對應收款截圖或銀行入帳。[cite:152]
- 有成本資料但尚未分類，可能造成淨利失真。[cite:147][cite:151]

定期盤點與稽核隱藏成本，被視為降低利潤流失的有效方式，因此警示頁是本產品的重要差異化功能。[cite:150][cite:154]

## 資料模型
建議先用統一事件模型，讓不同格式都能進入同一套欄位：

| 欄位 | 說明 |
|---|---|
| event_type | income / expense / payment / refund / tax |
| source_type | chat / bank / meta / logistics / packaging / tax_mail / excel / manual |
| source_file | 原始上傳檔名或來源 |
| date | 事件日期 |
| order_id | 訂單編號，可空白 |
| customer_name | 客戶名稱，可空白 |
| amount | 金額 |
| category | 收入或成本分類 |
| payment_status | 未確認 / 已確認 |
| match_status | 未配對 / 已配對 / 待人工確認 |
| note | 備註 |

這種設計可以先解決「先收進來」的問題，再逐步優化辨識與自動關聯能力。[cite:152][cite:155]

## 介面架構
### 首頁：本月淨利總覽
以 KPI 卡片 + 趨勢圖 + 待處理清單呈現，重點不是漂亮，而是讓老闆一打開就知道「有沒有賺、哪裡怪、今天要補什麼資料」。即時財務儀表板對小型商家特別重要，因為廣告與成本常會突然跳升。[cite:149][cite:151]

### 警示頁：隱藏成本
把所有異常集中成清單，例如「本月包材支出暴增 45%」、「有 12 筆收入未配對銀行入帳」、「Meta 廣告已吃掉營收 28%」。隱藏成本與分類缺漏會直接扭曲淨利，因此這頁應該是高頻使用頁面。[cite:147][cite:150][cite:151]

### 匯入頁：任意資料上傳
單一入口接住圖片、PDF、Excel、CSV、文字，並顯示辨識結果與待確認欄位。OCR 與欄位驗證被廣泛用來把發票、帳單與費用文件轉成結構化資料，適合作為此頁核心流程。[cite:152][cite:155]

### 資料頁：手動補登與編修
提供像簡化版試算表的列表與表單，讓使用者可修改辨識錯誤、補成本分類、補訂單號或備註。[cite:156]

## 優先順序
### 第一階段必做
- Excel 匯入
- 手動輸入
- 圖片/PDF 基礎辨識
- 收入/成本分類
- 本月淨利總覽
- 隱藏成本警示頁
- 銀行入帳手動配對

### 第二階段再做
- 自動銀行對帳
- 更強的 OCR 欄位辨識
- Meta/物流 API 串接
- 稅務文件自動分類
- 訂單與客戶維度報表

先從能解決核心痛點的 MVP 做起，符合 MVP 應專注於單一核心問題並快速驗證需求的原則。[cite:113]

## 成功指標
第一版可以觀察這些指標：

- 使用者完成第一次匯入並看到本月淨利所需時間
- 每月成功分類的成本筆數
- 未配對收入占比
- 未分類成本占比
- 本月淨利計算完整率
- 使用者每週回訪次數

若產品真的幫上忙，最直接的訊號不是功能用得多，而是「以前不知道自己賺多少，現在每月看得到真實淨利」。[cite:149][cite:151]

## 一句話版本
這個產品不是一般記帳軟體，而是專門幫「聊天成交型賣家」把訂單、收款證明、銀行入帳與四散成本整合起來，最後直接算出本月淨利，並主動指出隱藏成本的財務助手。[cite:147][cite:149][cite:151]
